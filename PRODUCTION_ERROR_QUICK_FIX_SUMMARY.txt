ISSUE
================================================================================
Error: TypeError: Cannot read properties of null (reading 'station_name')
User:   jmlambo@zinwa.co.zw (Harare SC - no stations registered)
Module: ClearWater → Production Data tab
Result: App crashes, user trapped on error page, navigation blocked

ROOT CAUSE
================================================================================
File: src/components/clearwater/ProductionDataTab.tsx
Line: 253-254

Code tried to access:
  record.stations.station_name

But record.stations was NULL (no station data available)

When station doesn't exist or can't be accessed:
  record.stations = null
  record.stations.station_name = ERROR (trying to read property of null)
  React ErrorBoundary catches error
  Error page shown
  Navigation broken (clicking TopBar causes same error again)

THE NATURE OF THE ERROR
================================================================================

Type:       Null Reference Exception (Runtime Data Error)
Severity:   HIGH - Blocks entire module
Trigger:    Attempt to read property from null object
Scope:      Any production log with missing/inaccessible station

Scenarios Where This Happens:
  1. Service centre has no registered stations (Harare SC case)
  2. Station was deleted after log was created (orphaned record)
  3. RLS policy prevents reading the station
  4. Data integrity issue

Why Navigation Was Blocked:
  User on error page
    ↓ clicks TopBar link (e.g., "CW")
    ↓ navigates to /clearwater
    ↓ ProductionDataTab renders again
    ↓ Same null error occurs
    ↓ Error page shown again
    ↓ User stuck in loop!

THE FIX
================================================================================

File:       src/components/clearwater/ProductionDataTab.tsx
Changes:    3 critical improvements

CHANGE #1: Null-Safe Property Access (Critical)
Location:   Lines 253-275

BEFORE (CRASHES):
  {logs.map((record) => (
    <tr>
      <td>
        {record.stations.station_name}  ← CRASH HERE
      </td>
    </tr>
  ))}

AFTER (SAFE):
  {logs.map((record) => {
    const stationName = record.stations?.station_name || 'Unknown Station';
    const stationCode = record.stations?.station_code || 'N/A';
    const isMissingData = !record.stations;

    return (
      <tr className={isMissingData ? 'bg-yellow-50' : ''}>
        <td>
          {stationName}
          {isMissingData && (
            <span className="bg-yellow-200 text-yellow-800">Missing</span>
          )}
        </td>
      </tr>
    );
  })}

Key: Optional chaining operator (?.)
  record.stations?.station_name
  Returns undefined (not null), prevents crash
  || 'Unknown Station' provides fallback value

CHANGE #2: Visual Indicator for Bad Data
Location:   Lines 268-272

Shows yellow "Missing" badge for logs with no station data
Highlights row with yellow background
User sees problem clearly without crash

CHANGE #3: Better Empty State Messages
Location:   Lines 221-229

Checks stations.length === 0 FIRST
Shows: "No stations registered" (not "No logs found")
Provides link to register stations
Clear guidance for user

EXPLICIT HOW TO FIX IT
================================================================================

Problem Code:
  {record.stations.station_name}

Breaks Because:
  - record.stations can be null
  - JavaScript throws error when reading property of null
  - React catches error, shows error page
  - Navigation blocked

Solution:
  {record.stations?.station_name || 'Unknown Station'}

How It Works:
  1. ?. (optional chaining) = "safely access if not null"
  2. record.stations?.station_name:
     - If stations exists: returns station_name
     - If stations is null: returns undefined (NOT an error!)
  3. || 'Unknown Station':
     - If undefined (from above): returns 'Unknown Station'
     - User sees: "Unknown Station" instead of crash

Result:
  - Component renders without error
  - User sees meaningful fallback value
  - No navigation blocking
  - No error page

   - No breaking changes

5. Maintainability
   - Clear intent (isMissingData variable)
   - Readable code (not convoluted null checks)
   - Easy to extend (can add more warnings)
