HOW IT WORKS
================================================================================

SCENARIO 1: User Logs In
─────────────────────────

Step 1: User enters credentials
Step 2: Authentication succeeds
Step 3: ProtectedRoute receives user + accessContext
Step 4: Detects user on root path "/"
Step 5: Looks up user's scope:
        • scopeType: 'SC'
        • scopeId: 'harare-id'
Step 6: Generates path: /sc/harare-id/dashboard
Step 7: Navigates to that path
Step 8: User lands on their authorized dashboard

✓ RESULT: User cannot land anywhere except their dashboard

─────────────────────────────────────────────────────────────────────────────

SCENARIO 2: User Tries Unauthorized URL
────────────────────────────────────────

User: jmlambo@zinwa.co.zw (Harare SC)
Authorized routes: /sc/harare-id/*
Unauthorized routes: /sc/murombedzi-id/*

Step 1: User manually types: /sc/murombedzi-id/dashboard
Step 2: RouteGuard intercepts route change
Step 3: Extracts from URL: routeSCId = 'murombedzi-id'
Step 4: Gets user's scopeId: 'harare-id'
Step 5: Compares: 'harare-id' !== 'murombedzi-id'
Step 6: Authorization: DENIED
Step 7: Logs security warning to console
Step 8: Redirects to: /sc/harare-id/dashboard
Step 9: Shows alert: "Access Denied: You are not authorized..."

✓ RESULT: User blocked from accessing Murombedzi data

─────────────────────────────────────────────────────────────────────────────

SCENARIO 3: Catchment User Accessing SC in Catchment
─────────────────────────────────────────────────────

User: Catchment Manager (Mashonaland West)
Scope: CATCHMENT (catchment-id)
Allowed SCs: [harare-id, chinhoyi-id, murombedzi-id, ...]

Step 1: User navigates to: /sc/harare-id/dashboard
Step 2: RouteGuard extracts: routeSCId = 'harare-id'
Step 3: Checks user's allowedServiceCentreIds
Step 4: Finds 'harare-id' in the list
Step 5: Authorization: GRANTED
Step 6: User accesses Harare SC dashboard

✓ RESULT: Catchment users can access SCs in their catchment

─────────────────────────────────────────────────────────────────────────────

SCENARIO 4: Non-Admin Tries Admin Route
────────────────────────────────────────

User: Technical Officer (system_rank: 50)

Step 1: User navigates to: /admin/users
Step 2: RouteGuard detects /admin prefix
Step 3: Checks: user.systemRank >= 70?
Step 4: Result: 50 < 70 → DENIED
Step 5: Redirects to user's dashboard
Step 6: Shows "Access Denied" alert

✓ RESULT: Admin routes protected by system rank
